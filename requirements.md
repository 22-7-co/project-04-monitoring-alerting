## 功能需求

### FR-1：指标采集与存储

#### FR-1.1：部署 Prometheus 进行指标采集

**说明**：将 Prometheus 搭建为中心化的指标采集与存储系统。

**要求**：
- 使用 Docker Compose 部署 Prometheus 2.47+
- 配置持久化存储，保留周期为 30 天
- 配置服务发现，自动识别采集目标
- 启用 remote write 用于长期存储（可选）
- 使用多个 Prometheus 实例实现高可用（高级）

**验收标准**：
- [ ] Prometheus 正常运行并可通过 `http://localhost:9090` 访问
- [ ] 指标按 30 天保留策略存储
- [ ] 时序数据无缺口
- [ ] 常见查询性能 < 1 秒
- [ ] 存储使用量在预期范围内

#### FR-1.2：配置服务发现

**说明**：无需手动配置即可自动发现监控目标。

**要求**：
- Kubernetes 服务发现（如使用 K8s）
- 基于文件的服务发现（用于静态目标）
- 基于 DNS 的服务发现
- 对发现的目标自动重打标签（relabel）

**验收标准**：
- [ ] 新服务可在 1 分钟内自动被发现
- [ ] 服务下线后可自动移除
- [ ] 所有目标都应用了正确标签
- [ ] 无孤立的抓取配置

#### FR-1.3：采集基础设施指标

**说明**：监控所有基础设施组件的系统级指标。

**需采集指标**：
- **CPU**：每核使用率、平均负载、steal time
- **内存**：总量、已用、可用、交换分区使用率
- **磁盘**：各挂载点使用率、I/O 操作、吞吐量
- **网络**：发送/接收字节数、丢包错误、TCP 连接数

**验收标准**：
- [ ] 所有节点都部署了 Node Exporter
- [ ] 所有系统指标均可在 Prometheus 中查看
- [ ] 指标采集间隔为每 15 秒
- [ ] 无缺失数据点

#### FR-1.4：采集应用指标

**说明**：跟踪应用层性能与行为指标。

**需采集指标**：
- **HTTP 请求**：按 method、endpoint、status code 统计请求总数
- **请求耗时**：带分位数（P50、P95、P99）的直方图
- **错误率**：按端点统计 4xx 与 5xx 错误
- **活跃连接数**：当前活跃连接数量
- **请求大小**：请求/响应体大小直方图

**验收标准**：
- [ ] 应用使用 `prometheus_client` 库完成埋点
- [ ] 所有服务都暴露 `/metrics` 端点
- [ ] 所有 HTTP 请求均被跟踪
- [ ] 延迟分桶（latency buckets）配置合理

#### FR-1.5：采集 ML 指标

**说明**：监控模型性能与健康状态相关的 ML 专属指标。

**需采集指标**：
- **每秒预测数**：按模型统计预测速率
- **推理延迟**：P50、P95、P99 推理耗时
- **模型准确率**：当前模型准确率（每日更新）
- **预测置信度**：置信度分布
- **数据漂移**：按特征统计漂移分数
- **缺失特征**：请求中缺失特征的数量

**验收标准**：
- [ ] 所有 ML 指标已在预测代码中完成埋点
- [ ] 指标可实时更新
- [ ] 可查看历史趋势
- [ ] 已集成漂移检测

### FR-4：告警与通知

#### FR-4.1：配置 Alertmanager

**说明**：搭建 Alertmanager 进行告警路由与通知。

**要求**：
- 部署 Alertmanager 0.26+
- 配置路由树
- 配置接收端（邮件、Slack、PagerDuty）
- 实现抑制规则（inhibition）
- 配置分组与限流

**验收标准**：
- [ ] Alertmanager 运行于 `http://localhost:9093`
- [ ] 所有路由规则工作正常
- [ ] 告警分组正确
- [ ] 去重机制生效

#### FR-4.2：基础设施告警规则

**说明**：定义基础设施问题告警。

**必需告警**：

1. **CPU 使用率过高**
   - **条件**：CPU > 80%，持续 5 分钟
   - **严重级别**：Warning
   - **处理动作**：排查负载，考虑扩容

2. **内存使用率过高**
   - **条件**：内存 > 85%，持续 5 分钟
   - **严重级别**：Warning
   - **处理动作**：检查内存泄漏，重启服务

3. **磁盘空间不足**
   - **条件**：磁盘可用空间 < 15%
   - **严重级别**：Critical
   - **处理动作**：清理日志，扩展存储

4. **服务不可用**
   - **条件**：目标 2 分钟不可达
   - **严重级别**：Critical
   - **处理动作**：重启服务，检查日志

**验收标准**：
- [ ] 所有告警规则已在 Prometheus 中配置
- [ ] 超过阈值时可触发告警
- [ ] 告警注释包含 runbook 链接
- [ ] 严重级别设置正确

#### FR-4.3：应用告警规则

**说明**：监控应用健康与性能。

**必需告警**：

1. **错误率过高**
   - **条件**：5xx 错误 > 5%，持续 5 分钟
   - **严重级别**：Critical
   - **处理动作**：查看日志，必要时回滚

2. **延迟过高**
   - **条件**：P95 延迟 > 1 秒，持续 5 分钟
   - **严重级别**：Warning
   - **处理动作**：分析应用性能，优化查询

3. **吞吐过低**
   - **条件**：请求速率 < 1 req/s，持续 10 分钟
   - **严重级别**：Info
   - **处理动作**：确认是否符合预期，排查流量下降

4. **响应时间过高**
   - **条件**：P99 延迟 > 5 秒
   - **严重级别**：Warning
   - **处理动作**：定位慢端点并优化

**验收标准**：
- [ ] 应用告警已配置
- [ ] 阈值已根据真实负载调优
- [ ] 测试中无明显误报
- [ ] 告警描述清晰

#### FR-4.4：ML 模型告警规则

**说明**：尽早检测 ML 专属问题。

**必需告警**：

1. **模型准确率下降**
   - **条件**：准确率 < 85%，持续 10 分钟
   - **严重级别**：Critical
   - **处理动作**：检查数据质量，考虑重新训练

2. **检测到数据漂移**
   - **条件**：漂移分数 > 0.5，持续 5 分钟
   - **严重级别**：Warning
   - **处理动作**：分析分布偏移，重训模型

3. **推理延迟过高**
   - **条件**：P99 推理 > 500ms，持续 5 分钟
   - **严重级别**：Warning
   - **处理动作**：优化模型，检查 GPU 利用率

4. **预测置信度过低**
   - **条件**：平均置信度 < 70%，持续 10 分钟
   - **严重级别**：Warning
   - **处理动作**：排查输入数据，检查模型版本

**验收标准**：
- [ ] ML 告警触发逻辑正确
- [ ] 已集成漂移检测
- [ ] 性能阈值设置合理
- [ ] Runbook 流程已文档化

#### FR-4.5：多渠道通知

**说明**：将告警路由到合适的通知渠道。

**通知渠道**：
- **邮件**：Info 和 Warning 告警
- **Slack**：Warning 和 Critical 告警

- **Webhook**：与自定义系统集成

```
Critical → PagerDuty + Slack
Warning → Slack + Email
Info → Email only
```

**验收标准**：
- [ ] 所有通知渠道均已配置
- [ ] 按严重级别路由正常
- [ ] 测试告警可成功送达
- [ ] 无重复通知

- 对重复告警进行限频（同一问题每小时最多 1 次）
- 在维护窗口期间静默告警

**验收标准**：
- [ ] 告警分组已配置
- [ ] 抑制规则工作正常
- [ ] 限频可防止告警刷屏
- [ ] 静默功能已测试

## 服务级别指标（SLIs）

### 应用 SLIs

| SLI | 定义 | 目标 | 测量方式 |
|-----|------|------|----------|
| **可用性** | 成功请求占比 | 99.9% | `(2xx+3xx requests) / total requests` |
| **延迟** | P95 响应时间 | < 200ms | `histogram_quantile(0.95, http_request_duration_seconds)` |
| **错误率** | 失败请求占比 | < 1% | `5xx requests / total requests * 100` |
| **吞吐量** | 每秒请求数 | > 100 | `rate(http_requests_total[5m])` |

### ML 模型 SLIs

| SLI | 定义 | 目标 | 测量方式 |
|-----|------|------|----------|
| **准确率** | 模型预测准确率 | > 90% | `correct_predictions / total_predictions` |
| **推理延迟** | P95 推理时间 | < 100ms | `histogram_quantile(0.95, model_inference_duration_seconds)` |
| **预测吞吐量** | 每秒预测数 | > 50 | `rate(model_predictions_total[5m])` |
| **漂移分数** | 数据分布漂移程度 | < 0.5 | `data_drift_score`（PSI 或 KS 统计量） |

### 基础设施 SLIs

| SLI | 定义 | 目标 | 测量方式 |
|-----|------|------|----------|
| **CPU 利用率** | 平均 CPU 使用率 | < 70% | `100 - avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100` |
| **内存利用率** | 平均内存使用率 | < 80% | `(1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100` |
| **磁盘利用率** | 磁盘使用率 | < 85% | `(node_filesystem_size_bytes - node_filesystem_avail_bytes) / node_filesystem_size_bytes * 100` |

## 服务级别目标（SLOs）

### 可用性 SLO

**目标**：99.9% 可用性（按月）

**允许**：
- 每月最多 43.2 分钟不可用
- 每年最多 8.64 小时不可用

**测量窗口**：滚动 30 天

**告警阈值**：7 天内可用性 < 99.95%（提前预警）

### 延迟 SLO

**目标**：95% 的请求在 < 200ms 内完成

**测量方式**：以 5 分钟窗口计算 P95 延迟

**告警阈值**：P95 > 250ms 持续 10 分钟

### 错误率 SLO

**目标**：错误率 < 1%

**测量方式**：5 分钟内 5xx 错误数 / 请求总数

**告警阈值**：错误率 > 2% 持续 5 分钟

### 模型准确率 SLO

**目标**：准确率 > 90%（每日）

**测量方式**：基于真实标签计算准确率

**告警阈值**：准确率 < 85% 持续 1 小时
