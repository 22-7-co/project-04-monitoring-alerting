## 系统概览

### 目的

该监控与告警系统为 ML 基础设施提供全面可观测性，支持：

- **实时可见性**：掌握系统健康状态与性能
- **主动告警**：在问题影响用户前提前预警
- **根因分析**：通过指标与日志关联定位问题
- **性能优化**：基于历史趋势持续改进
- **合规保障**：满足 SLA 与 SLO 要求

### 核心原则

1. **四大黄金信号**：监控延迟、流量、错误、饱和度
2. **USE 方法**：资源利用率、饱和度、错误
3. **RED 方法**：服务速率、错误、时长
4. **可行动告警**：每条告警都应触发明确的人类行动
5. **基于症状的告警**：优先对用户可见症状告警，而非仅内部指标

### 1. 指标采集层

```
┌────────────────────────────────────────────────────────────┐
│                     指标采集架构                           │
└────────────────────────────────────────────────────────────┘

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│     应用     │     │   基础设施    │     │   ML 模型    │
│              │     │   （节点）    │     │              │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                     │
       │ /metrics           │ /metrics            │ /metrics
       │                    │                     │
       └────────────────────┼─────────────────────┘
                            │
                ┌───────────▼──────────┐
                │      Prometheus      │
                │                      │
                │  ┌────────────────┐ │
                │  │ 抓取目标       │ │
                │  ├────────────────┤ │
                │  │ node-exporter  │ │
                │  │ ml-api:5000    │ │
                │  └────────────────┘ │
                │                      │
                │  ┌────────────────┐ │
                │  │ TSDB 存储      │ │
                │  │ （保留 30 天） │ │
                │  └────────────────┘ │
                │                      │
                │  ┌────────────────┐ │
                │  │ 告警规则       │ │
                │  └────────┬───────┘ │
                └───────────┼──────────┘
                            │
                ┌───────────▼──────────┐
                │    Alertmanager      │
                └──────────────────────┘
```

**组件说明**：

- **Prometheus Server**：时序数据库与抓取器
  - 每 15 秒从目标拉取一次指标
  - 本地存储数据，保留 30 天
  - 每 30 秒评估一次告警规则

- **Exporters**：以 Prometheus 格式暴露指标
  - `node_exporter`：系统指标（CPU、内存、磁盘）
  - `blackbox_exporter`：端点健康检查（可选）
  - 应用内置 `/metrics` 端点

- **服务发现（Service Discovery）**：自动检测采集目标
  - Kubernetes SD：发现 K8s 中的 Pod/Service
  - File SD：基于静态配置文件
  - DNS SD：通过 DNS 进行服务发现

**组件说明**：

- **Filebeat**：轻量日志传输代理
  - 监控日志文件变更
  - 添加元数据（主机名、标签）
  - 下游故障时本地缓冲
  - 发送到 Logstash 或 Elasticsearch

- **Logstash**：日志处理管道
  - 解析 JSON 日志
  - 过滤敏感数据
  - 增强上下文信息
  - 路由到 Elasticsearch

- **Elasticsearch**：分布式检索与分析引擎
  - 按天索引存储日志
  - 提供全文检索能力
  - 支持聚合分析
  - 通过 ILM 实现 90 天保留

- **Kibana**：可视化与探索
  - 检索并过滤日志
  - 构建可视化图表
  - 创建仪表盘
  - 保存检索条件

## 指标架构

### 指标类型与使用场景

```
┌─────────────────────────────────────────────────────────────┐
│                      指标类型选择                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Counter: 单调递增                                           │
│  ┌─────────────────────────────────────────┐               │
│  │ 用于：                                   │               │
│  │ - 请求总数                               │               │
│  │ - 错误总数                               │               │
│  │ - 预测总数                               │               │
│  │                                          │               │
│  │ 查询：rate() / increase()                │               │
│  │ 示例：rate(http_requests_total[5m])      │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
│  Gauge: 可增可减                                             │
│  ┌─────────────────────────────────────────┐               │
│  │ 用于：                                   │               │
│  │ - CPU 使用率                              │               │
│  │ - 内存使用率                              │               │
│  │ - 活跃连接数                              │               │
│  │ - 模型准确率                              │               │
│  │                                          │               │
│  │ 查询：直接查询或 avg()                   │               │
│  │ 示例：node_memory_usage_bytes            │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
│  Histogram: 值分布                                           │
│  ┌─────────────────────────────────────────┐               │
│  │ 用于：                                   │               │
│  │ - 请求耗时                               │               │
│  │ - 响应大小                               │               │
│  │ - 推理延迟                               │               │
│  │                                          │               │
│  │ 查询：histogram_quantile()               │               │
│  │ 示例：histogram_quantile(0.95, ...)      │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
│  Summary: 预计算分位数（不推荐）                             │
│  ┌─────────────────────────────────────────┐               │
│  │ 建议：新代码避免使用                     │               │
│  │ 推荐：使用 Histogram 更灵活              │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 指标命名规范

```
<metric_name>{label1="value1", label2="value2"}

最佳实践：
- 指标名使用 snake_case
- 名称中包含单位（bytes、seconds、total）
- 维度放在 labels 中，不放在指标名中
- 控制基数（唯一 label 组合建议 < 10k）

良好示例：
✓ http_request_duration_seconds{method="GET", endpoint="/predict"}
✓ model_predictions_total{model="resnet50", class="cat"}
✓ node_memory_usage_bytes{instance="server01"}

反例：
✗ httpRequestDurationSeconds (camelCase)
✗ requests (no unit, no context)
✗ http_get_predict_latency (dimensions in name, not labels)
✗ user_123_requests (high cardinality label)
```

### 网络安全

## 可扩展性考虑

### Prometheus 扩展

1. **纵向扩展**：增加内存与 CPU
2. **联邦（Federation）**：采用分层 Prometheus 架构
3. **Remote Write**：将长期数据写入 Thanos/Cortex
4. **分片（Sharding）**：将抓取目标拆分到多个实例

### Elasticsearch 扩展

1. **增加节点**：水平扩展到 3、5、7 个节点
2. **Hot-Warm-Cold 架构**：按数据年龄分层存储
3. **索引生命周期管理（ILM）**：自动管理索引
4. **分片调优**：优化分片大小（每分片 20-50GB）

### Grafana 扩展

1. **数据库后端**：使用 PostgreSQL 替代 SQLite
2. **缓存**：启用查询缓存
3. **负载均衡**：部署多个 Grafana 实例
4. **CDN**：通过 CDN 提供静态资源
